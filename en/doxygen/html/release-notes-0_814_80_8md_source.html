15 Upgrading and downgrading
16 =========================
17 
18 How to Upgrade
19 --------------
20 
21 If you are running an older version, shut it down. Wait until it has completely
22 shut down (which might take a few minutes for older versions), then run the
23 installer (on Windows) or just copy over /Applications/Dash-Qt (on Mac) or
24 dashd/dash-qt (on Linux). If you upgrade after DIP0003 activation and you were
25 using version < 0.13 you will have to reindex (start with -reindex-chainstate
26 or -reindex) to make sure your wallet has all the new data synced. Upgrading from
27 version 0.13 should not require any additional actions.
28 
29 Downgrade warning
30 -----------------
31 
32 ### Downgrade to a version < 0.13.0.0
33 
34 Downgrading to a version smaller than 0.13 is not supported anymore as DIP2/DIP3 has
35 activated on mainnet and testnet.
36 
37 ### Downgrade to versions 0.13.0.0 - 0.13.3.0
38 
39 Downgrading to 0.13 releases is fully supported until DIP0008 activation but is not
40 recommended unless you have some serious issues with version 0.14.

42 Notable changes
43 ===============
44 
45 DIP0004 - Coinbase Payload v2
46 -----------------------------
47 Coinbase Payload v2 introduces new field `merkleRootQuorums` which represents the merkle root of
48 all the hashes of the final quorum commitments of all active LLMQ sets. This allows SPV clients
49 to verify active LLMQ sets and use this information to further verify ChainLocks and LLMQ-based
50 InstantSend messages. Coinbase Payload v2 relies on DIP0008 (bit 4) activation.
51 
52 https://github.com/dashpay/dips/blob/master/dip-0004.md#calculating-the-merkle-root-of-the-active-llmqs
53 
54 DIP0008 - ChainLocks
55 --------------------
56 This version introduces ChainLocks, a technology for near-instant confirmation of blocks and
57 finding near-instant consensus on the longest valid/accepted chain. ChainLocks leverages LLMQ
58 Signing Requests/Sessions to accomplish this. ChainLocks relies on DIP0008 (bit 4) activation and
59 `SPORK_19_CHAINLOCKS_ENABLED` spork.
60 
61 Read more: https://github.com/dashpay/dips/blob/master/dip-0008.md
62 
63 DIP0010 - LLMQ-based InstantSend
64 --------------------------------
65 InstantSend is a feature to allow instant confirmations of payments. It works by locking transaction
66 inputs through masternode quorums. It has been present in Dash for a few years and been proven to work.
67 Nevertheless, there are some limits which could theoretically be removed in the old system but doing so
68 would have created risks in terms of scalability and security.
69 
70 We introduce LLMQ-based InstantSend which is designed to be much more scalable without sacrificing
71 security and which allows all transactions to be treated as InstantSend transactions. The old system
72 differentiated transactions as InstantSend transactions by using the P2P message “ix” instead of “tx”.
73 Since this distinction is not required in the new system, the P2P message “ix” will be removed after
74 DIP0008 deployment (for now, transactions sent via "ix" message will be relayed further via "tx" message).
75 
76 Read more: https://github.com/dashpay/dips/blob/master/dip-0010.md
77 
78 Network
79 ------
80 Legacy messages `mnw`, `mnwb`, `mnget`, `mnb`, `mnp`, `dseg`, `mnv`, `qdcommit` and their corresponding
81 inventory types (7, 10, 14, 15, 19, 22) are no longer suported.
82 
83 Message `version` is extended with a 256 bit field - a challenge sent to a masternode. Masternode which
84 received such a challenge must reply with new p2p message `mnauth` directly after `verack`. This `mnauth`
85 message must include a signed challenge that was previously sent via `version`.
86 
87 Mining
88 ------
89 Due to changes in coinbase payload this version requires for miners to signal their readiness via
90 BIP9-like mechanism - by setting bit 4 of the block version to 1. Note that if your mining software
91 simply uses `coinbase_payload` field from `getblocktemplate` RPC and doesn't construct coinbase payload
92 manually then there should be no changes to your mining software required. We however encourage pools
93 and solo-miners to check their software compatibility on testnet to ensure flawless migration.
94 
95 PrivateSend
96 -----------
97 The wallet will try to create and consume denoms a bit more accurately now. It will also only create a
98 limited number of inputs for each denominated amount to prevent bloating itself with mostly the smallest
99 denoms. You can control this number of inputs via new `-privatesenddenoms` cmd-line option (default is 300).
100 
101 InstantSend
102 -----------
103 Legacy InstantSend is going to be superseded by the newly implemented LLMQ-based one once DIP0008 (bit 4)
104 is active and `SPORK_20_INSTANTSEND_LLMQ_BASED` spork is ON.
105 
106 Sporks
107 ------
108 There are two new sporks introduced in this version - `SPORK_19_CHAINLOCKS_ENABLED` and
109 `SPORK_20_INSTANTSEND_LLMQ_BASED`. `SPORK_17_QUORUM_DKG_ENABLED` was introduced in v0.13 but was kept OFF.
110 It will be turned on once 80% masternodes are upgraded to v0.14 which will enable DKG and DKG-based PoSe.
111 
112 QR codes
113 --------
114 Wallet can now show QR codes for addresses in the address book, receiving addresses and addresses identified
115 in transactions list (right click -> "Show QR-code").
116 
117 RPC changes
118 -----------
119 There are a few changes in existing RPC interfaces in this release:
120 - for blockchain based RPC commands `instantlock` will say `true` if the transaction
121 was locked via LLMQ based ChainLocks (for backwards compatibility reasons)
122 - `prioritisetransaction` no longer allows adjusting priority
123 - `getgovernanceinfo` no longer has `masternodewatchdogmaxseconds` and `sentinelpingmaxseconds` fields
124 - `masternodelist` no longer supports `activeseconds`, `daemon`, `lastseen`, `protocol`, `keyid`, `rank`
125 and `sentinel` modes, new mode - `pubkeyoperator`
126 - `masternode count` no longer has `ps_compatible` and `qualify` fields and `ps` and `qualify` modes
127 - `masternode winner` and `masternode current` no longer have `protocol`, `lastseen` and `activeseconds`
128 fields, new field - `proTxHash`
129 - `debug` supports new categories: `chainlocks`, `llmq`, `llmq-dkg`, `llmq-sigs`
130 - `mnsync` no longer has `IsMasternodeListSynced` and `IsWinnersListSynced` fields
131 - various RPCs that had `instantlock` field have `chainlock` (excluding mempool RPCs) and
132 `instantlock_internal` fields now
133 
134 There are also new RPC commands:
135 - `bls fromsecret` parses a BLS secret key and returns the secret/public key pair
136 - `quorum` is a set of commands for quorums/LLMQs e.g. `list` to show active quorums (by default,
137 can specify different `count`) or `info` to shows detailed information about some specific quorum
138 etc., see `help quorum`
139 
140 Few RPC commands are no longer supported: `estimatepriority`, `estimatesmartpriority`,
141 `gobject getvotes`, `masternode start-*`, `masternode genkey`, `masternode list-conf`, `masternode check`,
142 `masternodebroadcast`, `sentinelping`
143 
144 See `help command` in rpc for more info.
145 
146 ZMQ changes
147 -----------
148 Added two new messages `hashchainlock` and `rawchainlock` which return the hash of the chainlocked block
149 or the raw block itself respectively.
150 
151 Command-line options
152 --------------------
153 
154 Changes in existing cmd-line options:
155 - `-bip9params` supports optional `window` and `threshold` values now
156 
157 New cmd-line options:
158 - `-watchquorums`
159 - `-privatesenddenoms`
160 - `-dip3params` (regtest-only)
161 - `-llmqchainlocks` (devnet-only)
162 
163 Few cmd-line options are no longer supported: `-limitfreerelay`, `-relaypriority`, `-blockprioritysize`,
164 `-sendfreetransactions`, `-mnconf`, `-mnconflock`, `-masternodeprivkey`
165 
166 See `Help -> Command-line options` in Qt wallet or `dashd --help` for more info.
167 
168 Miscellaneous
169 -------------
170 
171 A lot of refactoring, backports, code cleanups and other small fixes were done in this release.
