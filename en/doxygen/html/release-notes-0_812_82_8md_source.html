14 Upgrading and downgrading
15 =========================
16 
17 How to Upgrade
18 --------------
19 
20 If you are running an older version, shut it down. Wait until it has completely
21 shut down (which might take a few minutes for older versions), then run the
22 installer (on Windows) or just copy over /Applications/Dash-Qt (on Mac) or
23 dashd/dash-qt (on Linux).
24 
25 Downgrade warning
26 -----------------
27 ### Downgrade to a version < 0.12.2
28 
29 Because release 0.12.2 includes DIP0001 (2 MB block size hardfork) plus
30 a transaction fee reduction and a fix for masternode rank calculation algo
31 (which activation also depends on DIP0001) this release will not be
32 backwards compatible after DIP0001 lock in/activation happens.
33 
34 This does not affect wallet forward or backward compatibility.
35 
36 Notable changes
37 ===============
38 
39 DIP0001
40 -------
41 
42 We outline an initial scaling mechanism for Dash. After deployment and activation, Dash will be able to handle double the transactions it can currently handle. Together with the faster block times, Dash we will be prepared to handle eight times the traffic of Bitcoin.
43 
44 https://github.com/dashpay/dips/blob/master/dip-0001.md
45 
46 
47 Fee reduction
48 -------------
49 
50 All transaction fees are reduced 10x (from 10K per Kb to 1K per Kb), including fees for InstantSend (from 0.001 DASH per input to 0.0001 per input)
51 
52 InstantSend fix
53 ---------------
54 
55 The potential vulnerability found by Matt Robertson and Alexander Block was fixed in d7a8489f3 (#1620).
56 
57 RPC changes
58 -----------
59 
60 There are few changes in existing RPC in this release:
61 - There is no more `bcconfirmations` field in RPC output and `confirmations` shows blockchain only confirmations by default now. You can change this behaviour by switching new `addlockconf` param to `true`. There is a new rpc field `instantlock` which indicates whether a given transaction is locked via InstantSend. For more info and examples please see https://github.com/dashpay/dash/blob/v0.12.2.x/doc/instantsend.md;
62 - `gobject list` and `gobject diff` accept `funding`, `delete` and `endorsed` filtering options now, in addition to `valid` and `all` currently available;
63 - `vin` field in `masternode` commands is renamed to `outpoint` and shows data in short format now;
64 - `getblocktemplate` output is extended with versionbits-related information;
65 - Output of wallet-related commands `validateaddress` is extended with optional `hdkeypath` and `hdchainid` fields.
66 
67 There are few new RPC commands also:
68 - `masternodelist info` shows additional information about sentinel for each masternode in the list;
69 - `masternodelist pubkey` shows pubkey corresponding to masternodeprivkey for each masternode in the list;
70 - `gobject check` allows to check proposal data for correctness before preparing/submitting the proposal, `gobject prepare` and `gobject submit` should also perform additional validation now though;
71 - `setnetworkactive` allows to turn all network activity on and off;
72 - `dumphdinfo` displays some information about HD wallet (if available).
73

83 PrivateSend improvements
84 ------------------------
85 
86 Algorithm for selecting inputs was slightly changed in [`6067896ae`](https://github.com/dashpay/dash/commit/6067896ae) ([#1248](https://github.com/dashpay/dash/pull/1248)). This should allow user to get some mixed funds much faster.
87 
88 Lots of backports, refactoring and bug fixes
89 --------------------------------------------
90 
91 We backported some performance improvements from Bitcoin Core and aligned our codebase with their source a little bit better. We still do not have all the improvements so this work is going to be continued in next releases.
92 
93 A lot of refactoring and other fixes should make code more reliable and easier to review now.
94 
95 Experimental HD wallet
96 ----------------------
97 
98 This release includes experimental implementation of BIP39/BIP44 compatible HD wallet. Wallet type (HD or non-HD) is selected when wallet is created via `usehd` command-line option, default is `0` which means that a regular non-deterministic wallet is going to be used. If you decide to use HD wallet, you can also specify BIP39 mnemonic and mnemonic passphrase (see `mnemonic` and `mnemonicpassphrase` command-line options) but you can do so only on initial wallet creation and can't change these afterwards. If you don't specify them, mnemonic is going to be generated randomly and mnemonic passphrase is going to be just a blank string.
99 
100 **WARNING:** The way it's currently implemented is NOT safe and is NOT recommended to use on mainnet. Wallet is created unencrypted with mnemonic stored inside, so even if you encrypt it later there will be a short period of time when mnemonic is stored in plain text. This issue will be addressed in future releases.

383 Dash was previously known as Darkcoin.
384 
385 Darkcoin tree 0.8.x was a fork of Litecoin tree 0.8, original name was XCoin
386 which was first released on Jan/18/2014.
387 
388 Darkcoin tree 0.9.x was the open source implementation of masternodes based on
389 the 0.8.x tree and was first released on Mar/13/2014.
390 
391 Darkcoin tree 0.10.x used to be the closed source implementation of Darksend
392 which was released open source on Sep/25/2014.
393 
394 Dash Core tree 0.11.x was a fork of Bitcoin Core tree 0.9, Darkcoin was rebranded
395 to Dash.
396 
397 Dash Core tree 0.12.0.x was a fork of Bitcoin Core tree 0.10.
398 
399 Dash Core tree 0.12.1.x was a fork of Bitcoin Core tree 0.12.
