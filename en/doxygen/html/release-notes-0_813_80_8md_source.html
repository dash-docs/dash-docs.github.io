15 Upgrading and downgrading
16 =========================
17 
18 How to Upgrade
19 --------------
20 
21 If you are running an older version, shut it down. Wait until it has completely
22 shut down (which might take a few minutes for older versions), then run the
23 installer (on Windows) or just copy over /Applications/Dash-Qt (on Mac) or
24 dashd/dash-qt (on Linux). If you upgrade after DIP0003 activation you will
25 have to reindex (start with -reindex-chainstate or -reindex) to make sure
26 your wallet has all the new data synced.

37 Notable changes
38 ===============
39 
40 DIP0002 - Special Transactions
41 ------------------------------
42 Currently, new features and consensus mechanisms have to be implemented on top of the restrictions
43 imposed by the simple nature of transactions. Since classical transactions can only carry inputs
44 and outputs, they are most useful for financial transactions (i.e. transfers of quantities of Dash
45 between addresses on the distributed ledger). These inputs and outputs carry scripts and signatures
46 which are used to authorize and validate the transaction.
47 
48 To implement new on-chain features and consensus mechanisms which do not fit into this concept of
49 financial transactions, it is often necessary to misuse the inputs/outputs and their corresponding
50 scripts to add additional data and meaning to a transaction. For example, new opcodes would have
51 to be introduced to mark a transaction as something special and add a payload. In other cases,
52 OP_RETURN has been misused to store data on-chain.
53 
54 The introduction of special transactions will require the whole Dash ecosystem to perform a one-time
55 mandatory update of all the software and libraries involved. Software and libraries will have to be
56 changed so that they can differentiate between classical transactions and special transactions.
57 Deserialization of a classical transaction remains unchanged. Deserialization of a special transaction
58 requires the software/library to at least implement skipping and ignoring the extra_payload field.
59 Validation and processing of the inputs and outputs of a special transaction must remain identical to
60 classical transactions.
61 
62 Read more: https://github.com/dashpay/dips/blob/master/dip-0002.md
63 
64 DIP0003 - Deterministic Masternode Lists
65 ----------------------------------------
66 This DIP provides on-chain consensus for masternode lists that in turn allow for deterministic quorum
67 derivation and service scoring of masternode rewards.
68 
69 In the previous system, each node maintained its own individual masternode list. Masternodes gained
70 entry to that masternode list after the owner created a 1000 Dash UTXO and the masternode broadcast
71 a "masternode broadcast/announcement" P2P message. This in turn set the masternode to a PRE_ENABLED
72 state in the list maintained by each node. Masternodes then regularly broadcasted ping messages to
73 keep the masternode in ENABLED state.
74 
75 The previous system was maintained with consensus mechanisms that predated Satoshi Nakamoto’s solution
76 to the Byzantine Generals Problem. This meant that each node needed to maintain their own individual
77 masternode list with P2P messages and not a blockchain based solution. Due to the nature of the P2P
78 system, there was no guarantee that nodes would come to the same conclusion on what the masternode
79 list ought to look like. Discrepancies might, for example, occur due to a different order of message
80 reception or if messages had not been received at all. This posed some risks in regard to consensus
81 and limited the possible uses of quorums by the system.
82 
83 Additionally, the previous system required a complicated and failure prone "masternode sync" after
84 the initial startup of the node. After the blockchain was synced, the node would request the current
85 masternode list, the reward payment votes, and then verify the received list. This process tended to
86 take an unnecessarily long amount of time and sometimes resulted in failure.
87 
88 In the new system, the masternode list is derived entirely from information found on-chain. New
89 masternodes are added by new special transactions called Provider Registration Transactions
90 (abbreviated as ProRegTx). They are only removed by spending the collateral. A ProRegTx is a special
91 transaction which includes either a 1000-Dash collateral payment or a reference to it, along with
92 other payload information (DIP0002).
93 
94 The new system is going to be activated via combination of a BIP9-like deployment (bit 3) and new spork
95 (`SPORK_15_DETERMINISTIC_MNS_ENABLED`).
96 
97 Read more: https://github.com/dashpay/dips/blob/master/dip-0003.md
98 Upgrade instructions: https://docs.dash.org/DIP3-masternode-upgrade
99 
100 DIP0004 - Simplified Verification of Deterministic Masternode Lists
101 -------------------------------------------------------------------
102 A verifiable and correct masternode list is foundational to many Dash features, including verification
103 of an InstantSend transaction, mixing in PrivateSend and many features of Evolution. The deterministic
104 masternode lists introduced by DIP0003 enable full derivation and verification of a masternode list via
105 on-chain data. This, however, requires the full chain to be available to construct or verify this list.
106 A SPV client does not have the full chain and thus would have to rely on a list provided by one or more
107 nodes in the network. This provided list must be verifiable by the SPV client without needing the full
108 chain. This DIP proposes additions to the block’s coinbase transaction and new P2P messages to get and
109 update a masternode list with additional proof data.
110 
111 Read more: https://github.com/dashpay/dips/blob/master/dip-0004.md

113 Mining
114 ------
115 Please note that masternode payments in `getblocktemplate` rpc are now returned as an array and not as
116 a single object anymore. Make sure to apply corresponding changes to your pool software.
117 
118 Also, deterministic masternodes can now set their payout address to a P2SH address. The most common use
119 case for P2SH is multisig but script can be pretty much anything. If your pool software doesn't recognize
120 P2SH addresses, the simplest way to fix it is to use `script` field which shows scriptPubKey for each
121 entry of masternode payments array in `getblocktemplate`.
122 
123 And finally, after DIP0003 activation your pool software must be able to produce Coinbase Special
124 Transaction https://github.com/dashpay/dips/blob/master/dip-0004.md#coinbase-special-transaction.
125 Use `coinbase_payload` from `getblocktemplate` to get extra payload needed to construct this transaction.
126 
127 PrivateSend
128 -----------
129 With further refactoring of PrivateSend code it became possible to implement mixing in few parallel
130 mixing sessions at once from one single wallet. You can set number of mixing sessions via
131 `privatesendsessions` cmd-line option or dash.conf. You can pick any number of sessions between 1 and 10,
132 default is 4 which should be good enough for most users. For this feature to work you should also make
133 sure that `privatesendmultisession` is set to `1` via cmd-line or `Enable PrivateSend multi-session` is
134 enabled in GUI.
135 
136 Introducing parallel mixing sessions should speed mixing up which makes it reasonable to add a new
137 mixing denom (0.00100001 DASH) now while keeping all the old ones too. It also makes sense to allow more
138 mixing rounds now, so the new default number of rounds is 4 and the maximum number of rounds is 16 now.
139 
140 You can also adjust rounds and amount via `setprivatesendrounds` and `setprivatesendamount` RPC commands
141 which override corresponding cmd-line params (`privatesendrounds` and `privatesendamount` respectively).
142 
143 NOTE: Introducing the new denom and a couple of other changes made it incompatible with mixing on
144 masternodes running on pre-0.13 software. Please keep using 0.12.3 local wallet to mix your coins until
145 there is some significant number of masternodes running on version 0.13 to make sure you have enough
146 masternodes to choose from when the wallet picks one to mix funds on.
147 
148 InstantSend
149 -----------
150 With further improvements of networking code it's now possible to handle more load, so we are changing
151 InstantSend to be always-on for so called "simple txes" - transactions with 4 or less inputs. Such
152 transactions will be automatically locked even if they only pay minimal fee. According to stats, this
153 means that up to 90% of currently observed transactions will became automatically locked via InstantSend
154 with no additional cost to end users or any additional effort from wallet developers or other service
155 providers.
156 
157 This feature is going to be activated via combination of a BIP9-like deployment (we are reusing bit 3)
158 and new spork (`SPORK_16_INSTANTSEND_AUTOLOCKS`).
159 
160 Historically, InstantSend transactions were shown in GUI and RPC with more confirmations than regular ones,
161 which caused quite a bit of confusion. This will no longer be the case, instead we are going to show real
162 blockchain confirmations only and a separate indicator to show if transaction was locked via InstantSend
163 or not. For GUI it's color highlight and a new column, for RPC commands - `instantlock` field and `addlocked`
164 param.
165 
166 One of the issues with InstantSend adoption by SPV wallets (besides lack of Deterministic Masternode List)
167 was inability to filter all InstantSend messages the same way transactions are filtered. This should be
168 fixed now and SPV wallets should only get lock votes for transactions they are interested in.
169 
170 Another popular request was to preserve information about InstantSend locks between wallet restarts, which
171 is now implemented. This data is stored in a new cache file `instantsend.dat`. You can safely remove it,
172 if you don't need information about recent transaction locks for some reason (NOTE: make sure it's not one
173 of your wallets!).
174 
175 We also added new ZMQ notifications for double-spend attempts which try to override transactions locked
176 via InstantSend - `zmqpubrawinstantsenddoublespend` and `zmqpubhashinstantsenddoublespend`.
177 
178 Sporks
179 ------
180 There are a couple of new sporks introduced in this version `SPORK_15_DETERMINISTIC_MNS_ENABLED` (block
181 based) and `SPORK_16_INSTANTSEND_AUTOLOCKS` (timestamp based). There is aslo `SPORK_17_QUORUM_DKG_ENABLED`
182 (timestamp based) which is going to be used on testnet only for now.
183 
184 Spork data is stored in a new cache file (`sporks.dat`) now.
185 
186 Governance
187 ----------
188 Introduction of Deterministic Masternodes requires replacing of the old masternode private key which was used
189 both for operating a MN and for voting on proposals with a set of separate keys, preferably fresh new ones.
190 This means that votes casted for proposals by Masternode Owners via the old system will no longer be valid
191 after DIP0003 activation and must be re-casted using the new voting key.
192 
193 Also, you can now get notifications about governance objects or votes via new ZMQ notifications:
194 `zmqpubhashgovernancevote`, `zmqpubhashgovernanceobject`, `zmqpubrawgovernancevote` and
195 `zmqpubhashgovernanceobject`.

208 RPC changes
209 -----------
210 There are a few changes in existing RPC interfaces in this release:
211 - `gobject prepare` allows to send proposal transaction as an InstantSend one and also accepts an UTXO reference to spend;
212 - `masternode status` and `masternode list` show some DIP0003 related info now;
213 - `previousbits` and `coinbase_payload` fields were added in `getblocktemplate`;
214 - `getblocktemplate` now returns an array for masternode payments instead of a single object (miners and mining pools have to upgrade their software to support multiple masternode payees);
215 - masternode and superblock payments in `getblocktemplate` show payee scriptPubKey in `script` field in addition to payee address in `payee`;
216 - `getblockchaininfo` shows BIP9 deployment progress;
217 - `help command subCommand` should give detailed help for subcommands e.g. `help protx list`;
218 - `compressed` option in `masternode genkey`;
219 - `dumpwallet` shows info about dumped wallet and warns user about security issues;
220 - `instantlock` field added in output of `getrawmempool`, `getmempoolancestors`, `getmempooldescendants`, `getmempoolentry`,
221 `getrawtransaction`, `decoderawtransaction`, `gettransaction`, `listtransactions`, `listsinceblock`;
222 - `addlocked` param added to `getreceivedbyaddress`, `getreceivedbyaccount`, `getbalance`, `sendfrom`, `sendmany`,
223 `listreceivedbyaddress`, `listreceivedbyaccount`, `listaccounts`.
224 
225 There are also new RPC commands:
226 - `protx` (`list`, `info`, `diff`, `register`, `register_fund`, `register_prepare`,
227 `register_submit`, `update_service`, `update_registrar`, `revoke`);
228 - `bls generate`;
229 - `setprivatesendrounds`;
230 - `setprivatesendamount`.
