15 Upgrading and downgrading
16 =========================
17 
18 How to Upgrade
19 --------------
20 
21 If you are running an older version, shut it down. Wait until it has completely
22 shut down (which might take a few minutes for older versions), then run the
23 installer (on Windows) or just copy over /Applications/Dash-Qt (on Mac) or
24 dashd/dash-qt (on Linux). If you upgrade after DIP0003 activation and you were
25 using version < 0.13 you will have to reindex (start with -reindex-chainstate
26 or -reindex) to make sure your wallet has all the new data synced. Upgrading from
27 version 0.13 should not require any additional actions.
28 
29 When upgrading from a version prior to 0.14.0.3, the
30 first startup of Dash Core will run a migration process which can take a few minutes
31 to finish. After the migration, a downgrade to an older version is only possible with
32 a reindex (or reindex-chainstate).
33 
34 Downgrade warning
35 -----------------
36 
37 ### Downgrade to a version < 0.14.0.3
38 
39 Downgrading to a version smaller than 0.14.0.3 is not supported anymore due to changes
40 in the "evodb" database format. If you need to use an older version, you have to perform
41 a reindex or re-sync the whole chain.
42 
43 Notable changes
44 ===============
45 
46 Removal of the p2p alert system
47 -------------------------------
48 The p2p alert system was designed to send messages to all nodes supporting it by someone who holds
49 so called alert keys to notify such nodes in case of severe network issues. This version removes
50 the `alert` p2p message and `--alert` option. Internal alerts, partition detection warnings and the
51 `--alertnotify` option features remain.
52 
53 Removal of the legacy InstantSend system
54 ----------------------------------------
55 Version 0.14 introduced the new LLMQ-based InstantSend system which is designed to be much more scalable
56 than the legacy one without sacrificing security. The new system also allows all transactions to be treated as
57 InstantSend transactions. The legacy system was disabled together with the successful deployment of ChainLocks,
58 but we had to keep supporting the legacy system for a while to ensure a smooth transition period. This version finally drops
59 the legacy system completely.
60 
61 Read more about ChainLocks: https://github.com/dashpay/dips/blob/master/dip-0008.md
62 Read more about LLMQ-based InstantSend: https://github.com/dashpay/dips/blob/master/dip-0010.md
63 
64 Sporks
65 ------
66 The security level of ChainLocks and LLMQ-based InstantSend made sporks `SPORK_5_INSTANTSEND_MAX_VALUE` and
67 `SPORK_12_RECONSIDER_BLOCKS` obsolete, so they are removed now. Sporks `SPORK_15_DETERMINISTIC_MNS_ENABLED`,
68 `SPORK_16_INSTANTSEND_AUTOLOCKS` and `SPORK_20_INSTANTSEND_LLMQ_BASED` have no code logic behind them anymore
69 because they were used as part of the DIP0003, DIP0008 and DIP0010 activation process which is finished now.
70 They are still kept and relayed only to ensure smooth operation of v0.14 clients and will be removed in some
71 future version.
72 
73 Mempool sync improvements
74 -------------------------
75 Nodes joining the network will now try to sync their mempool from other v0.15+ peers via the `mempool` p2p message.
76 This behaviour can be disabled via the new `--syncmempool` option. Nodes serving such requests will now also push
77 `inv` p2p messages for InstandSend locks which are held for transactions in their mempool. These two changes
78 should help new nodes to quickly catchup on start and detect any potential double-spend as soon as possible.
79 This should also help wallets to slightly improve UX by showing the correct status of unconfirmed transactions
80 locked via InstandSend, if they were sent while the receiving wallet was offline. Note that bloom-filters
81 still apply to such `inv` messages, just like they do for transactions and locks that are relayed on a
82 regular basis.
83 
84 PrivateSend improvements
85 ------------------------
86 This version decouples the so called "Lite Mode" and client-side PrivateSend mixing, which allows client-side mixing
87 on pruned nodes running with `--litemode` option. Such nodes will have to also specify the newly redefined
88 `--enableprivatesend` option. Non-prunned nodes do not have to do this but they can use `--enableprivatesend`
89 option to disable mixing completely instead. Please note that specifying this option does not start mixing
90 automatically anymore (which was the case in previous versions). To automatically start mixing, use the new
91 `--privatesendautostart` option in addition to `--enableprivatesend`. Additionally, PrivateSend can always be
92 controlled with the `privatesend` RPC.
93 
94 Thanks to LLMQ-based InstantSend and its ability to lock chains of unconfirmed transactions (and not only a single
95 one like in the legacy system), PrivateSend mixing speed has improved significantly. In such an environment
96 Liquidity Provider Mode, which was introduced a long time ago to support mixing volume, is no longer needed and
97 is removed now. As such the `--liquidityprovider` option is not available anymore.
98 
99 Some other improvements were also introduced to speed up mixing, e.g. by joining more queues or dropping potential
100 malicious mixing participants faster by checking some rules earlier etc. Lots of related code was refactored to
101 further improve its readability, which should make it easier for someone to re-implement PrivateSend
102 correctly in other wallets if there is a desire to do so.
103 
104 Wallet changes
105 --------------
106 Wallet internals were optimized to significantly improve performance which should be especially notable for huge
107 wallets with tens of thousands of transactions or more. The GUI for such wallets should be much more responsive too now.
108 
109 Running Masternodes from local wallets was deprecated a long time ago and starting from this version we disable
110 wallet functionality on Masternodes completely.
111 
112 GUI changes
113 -----------
114 The Qt GUI went through a refresh to follow branding color guides and to make it feel lighter. All old themes besides
115 the Traditional one (the one with a minimal styling) were removed and instead a new Dark theme was added.
116 
117 In this version we made a lot of optimizations to remove various lags and lockups, the GUI in general should feel
118 much more smoother now, especially for huge wallets or when navigating through the masternode list. The latter has
119 a few new columns (collateral, owner and voting addresses) which give more options to filter and/or sort the list.
120 All issues with hi-dpi monitors should also be fixed now.
121 
122 The "Send" popup dialog was slightly tweaked to improve the language and provide a bit more information about inputs
123 included in the transaction, its size and the actual resulting fee rate. It will also show the number of inputs
124 a PrivateSend transaction is going to consume and display a warning regarding sender privacy if this number is 10
125 or higher.
126 
127 Changes in regtest and devnet p2p/rpc ports
128 -------------------------------------------
129 Default p2p and rpc ports for devnets and regtest were changed to ensure their consistency and to avoid
130 any potential interference with bitcoin's regtests. New default p2p/rpc ports for devnet are 19799/19798,
131 for regtest - 19899/19898 respectively.
132 
133 ZMQ changes
134 -----------
135 Added two new messages `rawchainlocksig` and `rawtxlocksig` which return the raw data of the block/transaction
136 concatenated with the corresponding `clsig`/`islock` message respectively.
137 
138 Crash reports and stack traces
139 ------------------------------
140 Binaries built with Gitian (including all official releases) will from now on always have crash reports and
141 crash hooks enabled. This means, that all binaries will print some information about the stack trace at the
142 time of a crash. If no debug information is present at crash time (which is usually the case), the binaries
143 will print a line that looks like this:
144 ```
145 2020-01-06 14:41:08 Windows Exception: EXCEPTION_ACCESS_VIOLATION
146 No debug information available for stacktrace. You should add debug information and then run:
147 dashd.exe -printcrashinfo=bvcgc43iinzgc43ijfxgm3yba....
148 ```
149 If you encounter such a crash, include these lines when you report the crash and we will be able to debug it
150 further. Anyone interested in running the specified `-printcrashinfo` command can do so after copying the debug info
151 file from the Gitian build to the same place where the binary is located. This will then print a detailed stack
152 trace.
153 
154 RPC changes
155 -----------
156 There are a few changes in existing RPC interfaces in this release:
157 - no more `instantsend` field in various RPC commands
158 - `use-IS`, `use_is` and `instantsend` options are deprecated in various RPC commands and have no effect anymore
159 - added new `merkleRootQuorums` field in `getblock` RPC results
160 - individual Dash-specific fields which were used to display soft-fork progress in `getblockchaininfo` are replaced
161  with the backported `statistics` object
162 - `privatesend_balance` field is shown in all related RPC results regardless of the Lite Mode or PrivateSend state
163 - added `pubKeyOperator` field for each masternode in `quorum info` RPC response
164 
165 There are also new RPC commands:
166 - `getbestchainlock`
167 - `getmerkleblocks`
168 - `getprivatesendinfo`
169 
170 `getpoolinfo` was deprecated in favor of `getprivatesendinfo` and no longer returns any data.
171 
172 There are also new RPC commands backported from Bitcoin Core 0.15:
173 - `abortrescan`
174 - `combinerawtransaction`
175 - `getblockstats`
176 - `getchaintxstats`
177 - `listwallets`
178 - `logging`
179 - `uptime`
180 
181 Make sure to check Bitcoin Core 0.15 release notes in a [section](#backports-from-bitcoin-core-015) below
182 for more RPC changes.
183 
184 See `help command` in rpc for more info.
185 
186 Command-line options
187 --------------------
188 Changes in existing cmd-line options:
189 - `--enableprivatesend` option has a new meaning now, see [PrivateSend](#privatesend) section for more info
190 
191 New cmd-line options:
192 - `--printcrashinfo`
193 - `--syncmempool`
194 - `--privatesendautostart`
195 
196 Few cmd-line options are no longer supported:
197 - `--alerts`
198 - `--masternode`, deprecated, specifying `--masternodeblsprivkey` option alone is enough to enable masternode mode now
199 - `--liquidityprovider`
200 - `--enableinstantsend`, dropped due to removal of the Legacy InstantSend
201 
202 Make sure to check Bitcoin Core 0.15 release notes in a [section](#backports-from-bitcoin-core-015) below
203 for more changes in command-line options.
204 
205 See `Help -> Command-line options` in Qt wallet or `dashd --help` for more info.
206 
207 Build system
208 ------------
209 This version always includes stacktraces in binaries now, `--enable-stacktraces` option is no longer available.
210 Instead you can choose if you want to hook crash reporting into various types of crashes by using `--enable-crash-hooks`
211 option (default is `no`). When using this option on macOS make sure to build binaries with `make -C src osx_debug`.
212 
213 Backports from Bitcoin Core 0.15
214 --------------------------------
215 
216 Most of the changes between Bitcoin Core 0.14 and Bitcoin Core 0.15 have been backported into Dash Core 0.15.
217 We only excluded backports which do not align with Dash, like SegWit or RBF related changes.
218 
219 You can read about changes brought by backporting from Bitcoin Core 0.15 in following docs:
220 - https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.15.0.md
221 - https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.15.1.md
222 
223 Some other individual PRs were backported from versions 0.16+, you can find the full list of backported PRs
224 and additional fixes in https://github.com/dashpay/dash/blob/master/doc/release-notes/dash/release-notes-0.15-backports.md
225 
226 Miscellaneous
227 -------------
228 A lot of refactoring, backports, code cleanups and other small fixes were done in this release. Dash-specific
229 modules were reorganized in separate folders to make navigation through code a bit easier.

487 Older releases
488 ==============
489 
490 Dash was previously known as Darkcoin.
491 
492 Darkcoin tree 0.8.x was a fork of Litecoin tree 0.8, original name was XCoin
493 which was first released on Jan/18/2014.
494 
495 Darkcoin tree 0.9.x was the open source implementation of masternodes based on
496 the 0.8.x tree and was first released on Mar/13/2014.
497 
498 Darkcoin tree 0.10.x used to be the closed source implementation of Darksend
499 which was released open source on Sep/25/2014.
500 
501 Dash Core tree 0.11.x was a fork of Bitcoin Core tree 0.9,
502 Darkcoin was rebranded to Dash.
503 
504 Dash Core tree 0.12.0.x was a fork of Bitcoin Core tree 0.10.
505 
506 Dash Core tree 0.12.1.x was a fork of Bitcoin Core tree 0.12.
