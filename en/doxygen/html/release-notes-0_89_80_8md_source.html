89 0.9.13.4 Release notes
90 ----------------------
91 
92 - Fixed issue with denominating small amounts of DRK in large wallet
93  (http://jira.darkcoin.qa/browse/DRK-46)
94 - Made splitting up initial inputs much more efficient. Now when splitting up,
95  it will use powers of two from 4096 DRK in reverse to get the best possible
96  mix of inputs for the next phase without any bloat to the blockchain or to
97  the users wallet.

99 - Stopeed collateral/fee creation when it should have been doing a full
100  split instead
101 - Sometimes the client would denominate less than the intended amount, then do
102  small denominations to make up the difference. This slowed down the
103  transactions and created extra transactions that weren't needed.
104 - Darksend should anonymize very close to the intended amount now
105 - Added Amount/Rounds to overview screen so you can see current settings
106 - Overview darksend cache is cleared on settings change (will instantly update)
107 - Fixed issue with completed amount jumping around
108  (http://jira.darkcoin.qa/browse/DRK-46)
109 - Made messages less threatening (http://jira.darkcoin.qa/browse/DRK-60)
110 
111 PS : Please move testing funds to a new wallet. This version has massive
112 optimizations for the way inputs are stored and split up. This will make

116 0.9.13.3 Release notes
117 ----------------------
118 
119 - This update deals with freezing, slow wallets, slow load times and the
120  "not compatible" error. I debugged one of the slow wallets and found it had
121  38,000 keys in the keypool, then after more investigation I found the passive
122  Darksend process has been reserve keys for every attempt! To rectify this
123  I've modified the queuing system, so users wait in a masternode queue without
124  actually sending the transactions until the queue is full.
125 
126 Please move any testing funds to a new wallet to test the new version.

366 0.9.12.10 Release notes
367 -----------------------
368 
369 Another huge update to the RC client, most of these are stability fixes and
370 anonymity improvements:
371 
372 - Removed "darksend denominate", darksend now will figure out the most it can
373  denominate. Use "darksend auto" instead.
374 - Fixed "Unknown State" display error
375 - Fixed 0.0025 collateral issues caused by issues in the state machine, you
376  should only be charged this amount now if you shutdown your client during
377  the Darksend process.
378 - Client will only submit 1 transaction into the pool fixing possible
379  anonymity issues
380 - Masternodes will only merge compatible transactions using the same
381  denominations. For example (500,500,100) would be able to merge
382  with (500,100), (10,1) with (10,1,1), but not (500,1) with (10,1).
383  This improves the anonmity by not allowing someone to follow transactions by
384  the missing denominations.
385 - Transactions use unique change addresses for every output of each round.
386 - QT GUI will now ask to unlock the wallet when it detects Darksend wants to do
387  something and lock it when it's done again.
388 - Darksend is turned off by default in the daemon now. In most cases daemons
389  won't want to run with anonymity (pools, exchanges, etc), if a user does they
390  can override the default setting with -enabledaemondarksend=true
391 - Fees per round of Darksend are 0.001DRK or $0.00538 at current prices. This
392  means to anonymize 1000DRK with 3 rounds (an average use case) it would cost
393  a user 1.5 cents.
394 - Protocol version is updated to kick old clients off testnet
