16 Upgrading and downgrading
17 =========================
18 
19 How to Upgrade
20 --------------
21 
22 If you are running an older version, shut it down. Wait until it has completely
23 shut down (which might take a few minutes for older versions), then run the
24 installer (on Windows) or just copy over /Applications/Dash-Qt (on Mac) or
25 dashd/dash-qt (on Linux).
26 
27 Downgrade warning
28 -----------------
29 
30 ### Downgrade to a version < 0.12.2.2
31 
32 Because release 0.12.2.2 included the [per-UTXO fix](release-notes/dash/release-notes-0.12.2.2.md#per-utxo-fix)
33 which changed the structure of the internal database, you will have to reindex
34 the database if you decide to use any pre-0.12.2.2 version.
35 
36 Wallet forward or backward compatibility was not affected.
37 
38 ### Downgrade to 0.12.2.2/3
39 
40 Downgrading to these versions does not require any additional actions, should be
41 fully compatible.
42 
43 Notable changes
44 ===============
45 
46 Introducing Named Devnets
47 -------------------------
48 
49 We introduce a new feature called [Named Devnets](https://github.com/dashpay/dash/pull/1791).
50 This feature allows the creation of multiple independent devnets. Each one is
51 identified by a name which is hardened into a "devnet genesis" block,
52 which is automatically positioned at height 1. Validation rules will
53 ensure that a node from `devnet=test1` will never be able to accept blocks
54 from `devnet=test2`. This is done by checking the expected devnet genesis
55 block.
56 
57 The genesis block of the devnet is the same as the one from regtest. This
58 starts the devnet with a very low difficulty, allowing us to fill up
59 needed balances for masternodes very fast.
60 
61 Also, the devnet name is put into the sub-version of the `VERSION` message.
62 If a node connects to the wrong network, it will immediately be disconnected.
63 
64 New format of network message signatures
65 ----------------------------------------
66 
67 We introduced a new signature format for Dash-specific network messages,
68 read more [here](https://github.com/dashpay/dash/pull/1936) and [here](https://github.com/dashpay/dash/pull/1937).
69 We also introduced a new spork `SPORK_6_NEW_SIGS` which is going to be used to activate the new format after the network has finished the upgrade.
70 Note that old pre-12.3 nodes won't be able to recognize and verify new signatures after `SPORK_6_NEW_SIGS` activates.
71 
72 The old format is partly kept in the code to keep backwards compatibility. This code will be removed in an upcoming
73 release.
74 
75 Governance system improvements
76 ------------------------------
77 
78 We do not use watchdogs since 12.2.x, instead we include all required information about sentinel into masternode
79 pings. With this update we add some additional information and cover everything with a signature to ensure that
80 masternode ping wasn't maleated by some intermediary node. All messages and logic related to watchdogs are
81 completely removed now. We also improved proposal message format, as well as proposal validation and processing,
82 which should lower network traffic and CPU usage. Handling of triggers was also improved slightly.
83 
84 `SPORK_13_OLD_SUPERBLOCK_FLAG` was removed now as it was unused since some time.
85 
86 PrivateSend improvements
87 ------------------------
88 
89 PrivateSend collaterals are no longer required to be N times of the PrivateSend fee (PSF), instead any input
90 which is greater or equal 1 PSF but less or equal 4 PSF can be used as a collateral. Inputs that are greater or
91 equal 1 PSF but strictly less than 2 PSF will be used in collaterals with OP_RETURN outputs. Note that such
92 inputs will be consumed completely, with no change outputs at all. This should lower number of inputs wallet
93 would need to take care of, improve privacy by eliminating the case where user accidentally merge small
94 non-private inputs together and also decrease global UTXO set size.
95 
96 It might feel that thanks to this change mixing fees are going to be slightly higher on average if have lots of
97 such small inputs. However, you need to keep in mind that creating new PrivateSend collaterals cost some fee too
98 and since such small inputs were not used at all you'd need more txes to create such collaterals. So in general,
99 we believe average mixing fees should stay mostly the same.
100 
101 There are also some other minor fixes which should also slightly improve mixing process.
102 
103 Additional indexes cover P2PK now
104 ---------------------------------
105 
106 Additional indexes like `addressindex` etc. process P2PK outputs correctly now. Note, that these indexes will
107 not be re-built automatically on wallet update, you must reindex manually to update indexes with P2PK outputs.
108 
109 Support for pruned nodes in Lite Mode
110 -------------------------------------
111 
112 It is now possible to run a pruned node which stores only some recent blocks and not the whole blockchain.
113 However this option is only available in so called Lite Mode. In this mode, Dash specific features are disabled, meaning
114 that such nodes won't fully validate the blockchain (masternode payments and superblocks).
115 PrivateSend and InstantSend functions are also disabled on such nodes. Such nodes are comparable to SPV-like nodes
116 in terms of security and validation - it relies a lot on surrounding nodes, so please keep this in mind if you decide to
117 use it for something.
118 
119 Default maximum block size
120 --------------------------
121 
122 We've changed the default maximum block size to 2MB. Such blocks were already allowed before, but the default setting
123 for the maximum block size (which only affects miners) was kept in place until this version.
124 
125 RPC changes
126 -----------
127 
128 There are a few changes in existing RPC interfaces in this release:
129 - `gobject count`, `masternode count` and `masternode list` will now by default return JSON formatted output;
130 If you rely on the old output format, you can still specify an additional parameter for backwards compatibility (`all` for `count` and `status` for `list`);
131 - `masternodelist` has a few new modes: `daemon`, `json`, `sentinel`;
132 - `debug` rpc now requires categories to be separated via `+`, not `,` like before (e.g. `dash+net`);
133 - `getchaintips` now shows the block fork occurred in `forkpoint` field;
134 - `getrawmempool`'s has InstantSend-related info (`instantsend` and `instantlock`);
135 - `getgovernanceinfo` has new field `sentinelpingmaxseconds`;
136 - `getwalletinfo` now shows PrivateSend balance in `privatesend_balance` field;
137 - `sendrawtransaction` no longer bypasses transaction policy limits by default.
138 - `dumphdinfo` should throw an error when wallet isn't HD now
139 
140 There is also a new RPC command `listaddressbalances`.
141 
142 You can read about RPC changes brought by backporting from Bitcoin Core in following docs:
143 - https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.13.0.md#low-level-rpc-changes
144 - https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.13.1.md#low-level-rpc-changes
145 - https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.14.0.md#low-level-rpc-changes
146 
147 Command-line options
148 --------------------
149 
150 New cmd-line options:
151 - introduced in Dash Core 0.12.3.1: `allowprivatenet`, `bip9params`, `sporkaddr`, `devnet`;
152 - backported from Bitcoin Core 0.13/0.14: `blockreconstructionextratxn`, `maxtimeadjustment`, `maxtipage`,
153 `incrementalrelayfee`, `dustrelayfee`, `blockmintxfee`.
154 
155 See `Help -> Command-line options` in Qt wallet or `dashd --help` for more info.
156 
157 New Masternode Information Dialog
158 ---------------------------------
159 
160 You can now double-click on your masternode in `My Masternodes` list on `Masternodes` tab to reveal the new
161 Masternode Information dialog. It will show you some basic information as well as software versions reported by the
162 masternode. There is also a QR code now which encodes corresponding masternode private key (the one you set with
163 mnprivkey during MN setup and NOT the one that controls the 1000 DASH collateral) which should make the process of pairing with
164 mobile software allowing you to vote with your masternode a bit easier (this software is still in development).
165 
166 Testnet fixes
167 -------------
168 
169 While we've been in release preparation, a miner used his ASICs on testnet. This resulted in too many blocks being mined
170 in a too short time. It revealed a few corner-case bugs in validation and synchronisation rules which we have fixed now.
171 We've also backported a special testnet rule for our difficulty adjustment algorithm that allows to mine a low difficulty
172 block on testnet when the last block is older than 5 minutes. This and the other fixes should stabilize our testnet in
173 case of future ASIC uses on testnet.
174 
175 Using masternode lists for initial peers discovery
176 --------------------------------------------------
177 
178 We now use a recent masternode list to feed the hardcoded seed nodes list in Dash Core. This list was previously
179 unmaintained as we fully relied on DNS based discovery on startup. DNS discovery is still used as the main discovery
180 method, but the hardcoded seed list should now be able to serve as a proper backup in case DNS fails for some reason.
181 
182 Lots of backports, refactoring and bug fixes
183 --------------------------------------------
184 
185 We backported many performance improvements and refactoring from Bitcoin Core and aligned most of our codebase with version 0.14.
186 Most notable ones besides various performance and stability improvements probably are
187 [Compact Block support (BIP 152)](https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.13.0.md#compact-block-support-bip-152),
188 [Mining transaction selection ("Child Pays For Parent")](https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.13.0.md#mining-transaction-selection-child-pays-for-parent),
189 [Null dummy soft fork (BIP 147, without SegWit)](https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.13.1.md#null-dummy-soft-fork),
190 [Nested RPC Commands in Debug Console](https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.14.0.md#nested-rpc-commands-in-debug-console) and
191 [Support for JSON-RPC Named Arguments](https://github.com/bitcoin/bitcoin/blob/master/doc/release-notes/release-notes-0.14.0.md#support-for-json-rpc-named-arguments).

200 
201 Note that some features were already backported earlier (per-UTXO fix, -assumevalid, GUI overlay etc.) and some were not backported at all
202 (SegWit and feefilter, you can read more about why we did so [here](https://blog.dash.org/segwit-lighting-rbf-in-dash-9536868ca861) and [here](https://github.com/dashpay/dash/pull/2025)).
203 The alert system was also kept in place for now. We are going to continue backporting the most notable fixes and improvements from Bitcoin Core versions 0.15 and 0.16 in future releases.
204 
205 A lot of refactoring, code cleanups and other small fixes were done in this release again. We are going to continue making code more reliable and easier to review in future releases as well.
